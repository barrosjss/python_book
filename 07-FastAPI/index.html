Sección 1

¿Qué es FastAPI?

Framework moderno para crear APIs REST con Python. Creado en 2018 por Sebastián Ramírez (Colombia 🇨🇴).

⚡ Muy rápido

Comparable a Node.js y Go. Uno de los más rápidos en Python.

📄 Docs automáticas

Genera Swagger UI y ReDoc sin escribir nada extra.

✅ Validación incluida

Pydantic valida los datos automáticamente. Menos bugs.

🐍 Python moderno

Usa type hints. El código es limpio y autodocumentado.

🌎
Lo usan

Netflix, Uber, Microsoft, NASA y miles de startups. Es el framework Python que más creció en los últimos años.

Sección 2

Instalación y Uvicorn

Instalar

$ pip install fastapi uvicorn

¿Qué es uvicorn? (la respuesta real)

FastAPI sabe construir una API, pero no sabe escuchar en un puerto de red. Uvicorn abre el puerto, recibe los bytes HTTP que llegan, los convierte en objetos Python, y se los pasa a FastAPI. FastAPI procesa y responde. Es separación de responsabilidades.

Técnicamente: FastAPI es un framework ASGI (Asynchronous Server Gateway Interface) y Uvicorn es el servidor ASGI que lo ejecuta. ASGI permite manejar miles de conexiones concurrentes, de ahí la velocidad.

Cómo ejecutar

$ uvicorn main:app --reload
# │ │ └── Recarga al guardar
# │ └── Variable 'app' en main.py
# └── Archivo main.py (sin .py)
🔄
dev vs producción

--reload solo para desarrollo (revisa cambios, más lento). En producción: uvicorn main:app --workers 4 para usar todos los núcleos del CPU.

Sección 3

¿Qué es un decorador?

main.py — anotado
from fastapi import FastAPI

app = FastAPI()          # Crear la "aplicación" (el servidor)

@app.get("/personajes")  # ← DECORADOR
#   │    └── La ruta URL que activa esta función
#   └── El método HTTP (GET, POST, PUT, DELETE)
def listar_personajes():
    # ↑ Esta función se ejecuta cuando llegue GET /personajes
    return [{"nombre": "Rick"}, {"nombre": "Morty"}]
    # Lo que retornes = JSON que recibe el cliente
🏷️
El decorador como etiqueta

El @app.get("/personajes") le dice a FastAPI: "cuando llegue una petición GET a la ruta /personajes, ejecuta la función de abajo". Es como poner un letrero en la puerta. Python permite que una función "envuelva" a otra — eso es un decorador.

⚠️
¿Por qué el decorador va ENCIMA de la función?

Porque en Python, @algo es azúcar sintáctica para funcion = algo(funcion). FastAPI registra tu función en su tabla de rutas en el momento en que Python la lee. Si no pones el decorador, la función existe pero FastAPI no sabe que debe llamarla cuando llegue una petición.

@app.get()

Leer datos

@app.post()

Crear datos

@app.put()

Actualizar

@app.delete()

Eliminar

FastAPI cuando terminas de escribir tu primer endpoint

¿Por qué se genera sola?

FastAPI lee tus decoradores, type hints y modelos Pydantic y construye un archivo JSON en /openapi.json siguiendo el estándar OpenAPI 3.0. Swagger UI lee ese JSON y genera la interfaz. Sin ese JSON, no habría docs.

🔌
OpenAPI = el contrato

El archivo /openapi.json es el "contrato" de tu API. Otros equipos (frontend, mobile) pueden importarlo en Postman o generar código cliente automáticamente a partir de él.

Sección 5

Tipo 1 — Path Parameters

Van dentro de la ruta. Identifican un recurso específico.

main.py
from fastapi import FastAPI

app = FastAPI()

@app.get("/personajes/{personaje_id}")
def obtener_personaje(personaje_id: int):
    #                              ↑ FastAPI convierte el texto de la URL a int
    return {
        "id": personaje_id,
        "mensaje": f"Buscando personaje con ID {personaje_id}"
    }

@app.get("/episodios/{temporada}/{numero}")
def obtener_episodio(temporada: int, numero: int):
    return {
        "temporada": temporada,
        "episodio": numero,
        "codigo": f"S0{temporada}E0{numero}"
    }
# GET localhost:8000/personajes/1
{"id": 1, "mensaje": "Buscando personaje con ID 1"}

# GET localhost:8000/episodios/3/7
{"temporada": 3, "episodio": 7, "codigo": "S03E07"}

Sección 6

Pydantic — Validación de datos

Define el molde de los datos y FastAPI valida automáticamente todo lo que recibe.

Definir un modelo

main.py
from fastapi import FastAPI
from pydantic import BaseModel
from typing import Optional

app = FastAPI()

# Modelo = molde de datos
class Estudiante(BaseModel):
    nombre: str           # obligatorio
    email: str            # obligatorio
    carrera: str          # obligatorio
    semestre: int         # obligatorio
    promedio: Optional[float] = None  # opcional

@app.post("/estudiantes")
def crear_estudiante(est: Estudiante):
    # FastAPI ya validó que est tiene todos los campos
    return {"mensaje": "creado", "datos": est}

Qué valida automáticamente

EnvíasResultado
Todos los campos correctos✅ 200/201 OK
Falta un campo obligatorio❌ 422 Validation Error
semestre: "cuatro"❌ 422 "not a valid integer"
Campo extra desconocido✅ Se ignora silenciosamente
🛡️
Sin Pydantic tendrías que escribir esto

if not nombre: raise ValueError(...)
if not isinstance(semestre, int): raise ValueError(...)
if not 1 <= semestre <= 12: raise ValueError(...)
Para cada campo. Pydantic lo hace en una línea: semestre: int.

🔄
Pydantic también convierte tipos

Si envías semestre: "4" (string), Pydantic convierte a 4 (int) si puede. Si envías semestre: "cuatro" → 422 error. Es inteligente, no estricto.

Sección 5

Tipo 3 — Request Body

Se usa en POST, PUT, PATCH. Los datos viajan dentro de la petición, no en la URL.

💡
¿Cuándo usar cada parámetro?

Path param: identificar cuál recurso (/estudiantes/5) · Query param: filtrar o paginar (?carrera=sistemas) · Body: enviar datos complejos en POST/PUT

main.py — los 3 tipos juntos
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class EstudianteUpdate(BaseModel):
    nombre: str
    promedio: float

# Path param: {estudiante_id}
# Query param: ?notificar=true
# Body: EstudianteUpdate (JSON en el cuerpo)
@app.put("/estudiantes/{estudiante_id}")
def actualizar_estudiante(
    estudiante_id: int,           # path param
    notificar: bool = False,      # query param
    datos: EstudianteUpdate = ... # body
):
    return {
        "id": estudiante_id,
        "notificado": notificar,
        "nuevo_nombre": datos.nombre
    }
Sección 7

HTTPException — Manejo de errores

Cuando algo sale mal, responde con el código HTTP correcto.

main.py
from fastapi import FastAPI, HTTPException

app = FastAPI()

estudiantes_db = {1: "Ana", 2: "Carlos", 3: "María"}

@app.get("/estudiantes/{est_id}")
def obtener_estudiante(est_id: int):
    if est_id not in estudiantes_db:
        raise HTTPException(
            status_code=404,
            detail=f"Estudiante {est_id} no encontrado"
        )
    return {"id": est_id, "nombre": estudiantes_db[est_id]}


# Respuesta cuando el ID no existe:
# {
#   "detail": "Estudiante 99 no encontrado"
# }
# Status: 404 Not Found
🎯
Patrón estándar

Siempre verifica si el recurso existe antes de operar. Si no existe → raise HTTPException(status_code=404, ...). El raise detiene la función inmediatamente y FastAPI responde con el error.

💡
¿Por qué importa el código correcto?

El cliente (requests, Streamlit, el navegador) toma decisiones según el código. 404 → "no encontrado, quizás el ID está mal". 500 → "error del servidor, reintenta". Si mandas 200 para todo, el cliente no puede saber qué pasó.

Sección 7

CRUD — GET y POST

main.py — Create + Read
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import Optional

app = FastAPI(title="API Estudiantes")

class Estudiante(BaseModel):
    nombre: str; email: str; carrera: str; semestre: int
    promedio: Optional[float] = None

class EstudianteConID(Estudiante):
    id: int

db: dict[int, EstudianteConID] = {}
contador = 0

# READ ALL
@app.get("/estudiantes", response_model=list[EstudianteConID])
def listar(carrera: Optional[str] = None):
    result = list(db.values())
    if carrera:
        result = [e for e in result if e.carrera.lower() == carrera.lower()]
    return result

# CREATE
@app.post("/estudiantes", response_model=EstudianteConID, status_code=201)
def crear(est: Estudiante):
    global contador
    contador += 1
    nuevo = EstudianteConID(id=contador, **est.dict())
    db[contador] = nuevo
    return nuevo

CRUD — GET por ID, PUT y DELETE

main.py — Read One + Update + Delete
# READ ONE
@app.get("/estudiantes/{est_id}", response_model=EstudianteConID)
def obtener(est_id: int):
    if est_id not in db:
        raise HTTPException(status_code=404, detail=f"Estudiante {est_id} no encontrado")
    return db[est_id]

# UPDATE
@app.put("/estudiantes/{est_id}", response_model=EstudianteConID)
def actualizar(est_id: int, datos: Estudiante):
    if est_id not in db:
        raise HTTPException(status_code=404, detail=f"Estudiante {est_id} no encontrado")
    actualizado = EstudianteConID(id=est_id, **datos.dict())
    db[est_id] = actualizado
    return actualizado

# DELETE
@app.delete("/estudiantes/{est_id}")
def eliminar(est_id: int):
    if est_id not in db:
        raise HTTPException(status_code=404, detail=f"Estudiante {est_id} no encontrado")
    eliminado = db.pop(est_id)
    return {"mensaje": f"'{eliminado.nombre}' eliminado", "id": est_id}
